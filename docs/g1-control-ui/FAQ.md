## 1. Доступность API и бэкенда

**Как понять, что бэкенд доступен?**  
На **Главной** отображается статус «API доступен». При недоступности показывается баннер вверху страницы и кнопка «Повторить» для повторной проверки `GET /api/health`.

**Что такое `VITE_API_BASE`?**  
Переменная окружения **сборки** фронтенда: базовый URL, на который уходят все запросы к REST и от которого строится WebSocket. Если не задана, браузер использует тот же origin, что и страница (в разработке Vite **проксирует** `/api` и `/ws` на машину с FastAPI). Эффективное значение видно в **Настройках** (`/settings`).

**Зачем `X-User-Id` и где его настроить?**  
Для маршрутов **Phase 5** (`/api/platform/*`, `/api/jobs*`, `/api/packages*`) бэкенд определяет владельца ресурсов по заголовку `X-User-Id`. Эффективное значение показано в **Настройках**: можно задать переопределение в браузере (localStorage), либо использовать **`VITE_PLATFORM_USER_ID`** при сборке, либо встроенный fallback (`local-dev`). HTTP-вызовы Phase 5 в коде должны идти через **`apiFetch`** в `src/api/client.ts`.

## 2. Авторинг: keyframes, motion, scenario

**Что редактируется на экране «Авторинг»?**  
Три типа артефактов Phase 0: **keyframes**, **motion**, **scenario** — как JSON. Есть загрузка golden-фикстур из `public/fixtures/golden/v1/` и кнопки форматирования/копирования.

**Где описаны форматы?**  
В репозитории бэкенда — каталог контрактов (см. [backend_references.md](backend_references.md)). Копии JSON Schema для офлайн-проверки лежат в `web/frontend/public/contracts/authoring/`.

## 3. Клиентская и серверная валидация

**Две кнопки проверки — в чём разница?**  
- **JSON Schema (клиент)** — проверка в браузере по локальной копии схемы (AJV). Быстро, работает без сервера.  
- **Сервер Phase 0** — `POST /api/validate`: источник правды для продакшн-логики на бэкенде.

**Почему результаты могут различаться?**  
Схема на клиенте могла устареть относительно сервера. После обновления бэкенда скопируйте актуальные `.schema.json` в `public/contracts/` и пересоберите фронт.

## 4. Pose Studio и ассеты

**Что за вкладка «WASM» / 3D в Pose Studio?**  
Клиентская симуляция через официальный пакет [`@mujoco/mujoco`](https://www.npmjs.com/package/@mujoco/mujoco) и MJCF робота **Unitree G1** из репозитория [mujoco_menagerie](https://github.com/google-deepmind/mujoco_menagerie/tree/main/unitree_g1). Модель и меши лежат в `public/mujoco/g1/`; для первичной загрузки ассетов разработчик запускает `npm run fetch:menagerie-g1`.

**Чем углы из WASM отличаются от потока телеметрии на бэкенде?**  
Бэкенд отдаёт поток **`/ws/telemetry`** (mock или DDS) для интеграций; в **текущей версии SPA** отдельного экрана с таблицей/слайдерами по этому потоку нет. **Pose Studio** использует **локальный** MuJoCo WASM и **`GET /api/joints`** для групп и подписей суставов; 3D и слайдеры WASM не подключаются к WebSocket. Экспорт keyframes в Авторинг использует индексы Phase 0 в поле `joints_deg` (`"0"`…`"28"`, значения в градусах).

**Несколько поз и «Создать движение» (вкладка 3D)?**  
Можно сохранить до **трёх дополнительных снимков** (кнопка «Добавить позу») поверх **текущей** позы в симуляторе — в экспорте Phase 0 получается до **четырёх keyframes** (текущая + снимки). Кнопка **«Скачать pose.json (SDK)»** выгружает JSON-массив в формате legacy `mid_level_motions/.../pose.json` (те же углы в градусах), совместимый с `execute.py`, сгенерированным через `action_exporter.save_action` в репозитории платформы. **«Создать движение»** проигрывает в окне 3D плавный переход от текущего состояния к сохранённым позам: интерполяция углов (cosine ease, длительность сегментов) в [`web/frontend/src/lib/motionInterpolation.ts`](../../web/frontend/src/lib/motionInterpolation.ts), затем `mj_forward`; это не полноценный шаг физики `mj_step` и не DDS-команды на робота.

**Что изменилось в продвинутом NLA-таймлайне Pose Studio?**  
Вместо только линейного списка поз теперь доступен multi-track редактор (`Hands`, `Legs`, `Torso`) с keyframes и ручками Безье (`in/out dt,dv`) на треке. Это позволяет:
- сглаживать шумные участки после freemocap/камерной записи (кнопка `Smooth range` на выбранном суставе);
- точечно править фазу переноса веса (через time/Bezier handles по конкретному суставу);
- отдельно включать/выключать треки и менять их вес при микшировании.

**Как это влияет на экспорт в keyframes и pose.json?**  
Экспорт остаётся обратно совместимым: NLA-таймлайн семплируется в обычный массив Phase 0 `keyframes` (с `timestamp_s` и `joints_deg`) и в legacy `pose.json` для SDK. То есть downstream пайплайн (`validate/preprocess/playback/train`) не требует нового формата входа.

**Почему первый заход в WASM «тяжёлый»?**  
Браузер один раз загружает WASM-рантайм MuJoCo (порядка мегабайт) и файлы сцены/STL. Их имеет смысл кэшировать на CDN/reverse-proxy.

Подробнее: [../mujoco-wasm-browser.md](../mujoco-wasm-browser.md).

## 5. Телеметрия и WebSocket

**Есть ли отдельный экран «Телеметрия»?**  
Нет. Маршрут **`/telemetry`** выполняет **редирект на `/pose`** (совместимость со старыми закладками). Отдельной страницы с live-потоком углов из `/ws/telemetry` в текущей сборке нет; хук [`useTelemetryWebSocket`](../../web/frontend/src/hooks/useTelemetryWebSocket.ts) в репозитории есть, но к экранам не подключён.

**Как узнать режим телеметрии на сервере?**  
При доступном API на **Главной** рядом со статусом показывается **`telemetry_mode`** из **`GET /api/meta`** (например `mock` или `dds`).

**Что значит режим DDS (`telemetry_mode: dds`)?**  
Бэкенд нацелен на поток телеметрии через DDS. Если DDS-мост не развёрнут, **`/ws/telemetry`** может сразу закрываться или нестабильно переподключаться — для обычной разработки UI переключите сервер на **mock-телеметрию** (см. README/OpenAPI бэкенда). Отдельного предупреждающего баннера на Pose Studio при `dds` во фронте **сейчас не показывается**; ориентируйтесь на строку режима на Главной и на справку бэкенда.

**Интеграции с `/ws/telemetry`**  
Для клиентов вне текущего SPA: тот же хост/схема, что и для REST (`VITE_API_BASE` или origin страницы); путь WebSocket согласован с базой API. При проблемах с соединением проверьте CORS/proxy для **`/ws`** на стороне reverse-proxy.

## 7. Конвейер и CLI

**Зачем экран показывает результат `GET /api/pipeline/detect-cli`?**  
Бэкенд проверяет, видны ли в `PATH` команды `preprocess` / `playback` / `train`. Если нет — возможен fallback через `python -m ...`.

**Где смотреть ошибки preprocess / playback / train?**  
В блоках вывода на экране «Конвейер»: `exit_code`, stdout, stderr; длинные JSON могут быть усечены с кнопкой «Показать полностью».

## 8. Платформа: артефакты и очередь train

**Зачем сохранять артефакты через Phase 5?**  
Крупный JSON (например `reference_trajectory` после preprocess) удобнее один раз сохранить на платформе под именем (`POST /api/platform/artifacts/{name}`) и ссылаться на него из `POST /api/jobs/train`.

**Чем конвейерный train отличается от задачи в очереди?**  
**Конвейер** запускает train как синхронный subprocess (`POST /api/pipeline/train`) — результат сразу в ответе. **Задачи** — асинхронная очередь (`POST /api/jobs/train`): изоляция workspace и лимиты параллелизма задаются на бэкенде.

**Как перенести reference из preprocess в очередь train?**  
После успешного preprocess на экране **Конвейер** нажмите «Перейти к очереди обучения» — откроется `/jobs` с предзаполненным `reference_trajectory`. Постановка в очередь **не** выполняется автоматически.

**Задача не выходит из очереди — что проверить?**  
На бэкенде должен работать обработчик очереди (часто переменная `G1_PLATFORM_WORKER_ENABLED`). Без worker задачи не выполняются.

**Как выпустить Skill Bundle после успешного train?**  
На экране деталей задачи (`/jobs/:jobId`), когда статус успешный, доступна кнопка упаковки — запрос `POST /api/packages/from-job/{job_id}`.

**Где список пакетов, скачивание и загрузка?**  
Экран **Пакеты** (`/packages`): `GET /api/packages` (таблица), `GET /api/packages/{package_id}/download` (архив), `POST /api/packages/upload` (multipart `.tar.gz`).

## 9. Продакшен и развёртывание

**Чем dev отличается от продакшена по CORS?**  
В разработке бэкенд часто разрешает все origins (`allow_origins=["*"]`). В продакшене список нужно сузить до доверенных хостов UI. Надёжная схема — отдавать SPA и проксировать `/api` и `/ws` с одного origin (reverse-proxy).

**Нужны ли `credentials` во фронтенде?**  
Текущий клиент вызывает `fetch` без `credentials: 'include'`. Phase 5 опирается на заголовок `X-User-Id`, а не на cookie-сессию.

**Где полный чеклист?**  
Файл [../deployment/README.md](../deployment/README.md) — чеклист и инструкция по Vercel.

## 11. Motion Studio: камера и pipeline

**Какой endpoint использует камера в Pose Studio?**  
`WS /ws/capture` обслуживается отдельным motion-capture сервисом (обычно `:8001`), не FastAPI на `:8000`. Для фронтенда endpoint задаётся через `VITE_MOTION_CAPTURE_WS_URL`.

**Что происходит, если `WS /ws/capture` недоступен?**  
`MotionCapturePanel` переключается на локальный fallback: браузерный `PoseLandmarker` определяет landmarks, а ретаргетинг продолжается через `POST /api/pipeline/retarget`. В статусе панели показывается, что используется local fallback.

**Зачем нужна 3-секундная T-pose калибровка?**  
Перед подключением live track UI запускает короткий отсчёт для стабилизации масштаба/центра позы и более устойчивого старта трекинга.

**Почему в Pose Studio видно предупреждение при `telemetry_mode=dds`?**  
Это защита от ложного ожидания «живой» DDS-телеметрии: если DDS-мост не поднят, часть функциональности может работать в mock/fallback-режиме.

**Что сохраняется после `Stop recording`?**  
UI сохраняет capture artifact на платформу (`POST /api/platform/artifacts/{name}`) в формате `aurosy_capture_v1` с полями `frames` (`[N,33,3]`) и `bvh`. Этот же `X-User-Id` используется и для pipeline, поэтому артефакт доступен в том же прогоне.

**Как из записи строится референс?**  
Базовый путь в UI: поле **Landmarks artifact** + кнопка «Собрать референс из landmarks» (`action: build_reference`, `landmarks_artifact`). На бэкенде также поддержаны `capture_artifact` и `bvh_artifact` для `build_reference`.

**Как восстановиться после падения train/package?**  
Используйте тот же `pipeline_id` (хранится в `sessionStorage` как `g1_motion_pipeline_id`), нажмите `Sync` в панели, при необходимости повторите `enqueue_train`/`request_pack` с `force: true` через API.

## 12. Live Mode troubleshooting

**Камера открывается, но в превью нет скелета**  
Проверьте права камеры в браузере и наличие `@mediapipe/tasks-vision` в зависимостях фронтенда.

**`/api/pipeline/retarget` отвечает ошибкой 400/500**  
Проверьте shape landmarks (`[33,3]` или `[N,33,3]`), доступность `joint_map`, совместимость `source_skeleton/target_robot`.

**Сильный jitter в live-позе**  
Проверьте FPS камеры и сетевую задержку; сервер применяет EMA-сглаживание для последовательностей, а UI дропает промежуточные кадры при перегрузке канала.
